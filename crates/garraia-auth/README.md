# garraia-auth

Authentication and authorization for the GarraIA Group Workspace.

> **Status:** verify path real (GAR-391a + GAR-391b). Axum extractor +
> `RequirePermission` + refresh/logout/signup endpoints + cross-group authz
> suite arrive in GAR-391c/d. See
> [`docs/adr/0005-identity-provider.md`](../../docs/adr/0005-identity-provider.md)
> for the normative design.

## What ships in 391a

- `IdentityProvider` trait — frozen shape, four methods.
- `InternalProvider` struct — stub bodies (replaced by 391b real impl).
- `LoginPool` newtype — dedicated `garraia_login` BYPASSRLS pool, validated at
  construction time via `SELECT current_user`. **Cannot** be built from a raw
  `PgPool`; the boundary is enforced at compile time.
- `Principal`, `Identity`, `Credential`, `RequestCtx`, `AuthError` types.
- Migration `008_login_role.sql` (in `garraia-workspace`) creating the
  `garraia_login NOLOGIN BYPASSRLS` role with the four exact GRANTs from
  ADR 0005 §"Login role specification".

## What ships in 391b

- **Real `InternalProvider::verify_credential`** with single-tx flow:
  `BEGIN → SELECT ... FOR NO KEY UPDATE OF ui → dispatch by hash prefix →
  audit insert → COMMIT`. Empirical race regression (`tests/concurrent_upgrade.rs`)
  validates the row lock prevents duplicate upgrade events.
- **Argon2id** new writes (`hash_argon2id`) using RFC 9106 first
  recommendation params (`m=64MiB, t=3, p=4`). PBKDF2-SHA256 verify-only
  for legacy hashes, with lazy upgrade in the same transaction as a
  successful verify.
- **Constant-time anti-enumeration** via `DUMMY_HASH` (Argon2id PHC over a
  discarded random secret, generated by `build.rs` at compile time).
  Consumed on every `user_not_found` and `account_not_active` terminal so
  the wall-clock latency matches a real verify.
- **Migration 009** (`user_identities.hash_upgraded_at`) — prereq for the
  lazy upgrade UPDATE, shipped separately via plan `0011.5` to honor the
  "structural fixes don't ride with feature impl" rule.
- **`audit_events`** in every login terminal: `login.success`,
  `login.failure_user_not_found` (actor_user_id NULL),
  `login.failure_wrong_password`, `login.failure_account_suspended`
  (covers `users.status='deleted'` too), `login.failure_unknown_hash`,
  `login.password_hash_upgraded`. Insert is inside the verify tx for v1
  atomicity. `actor_label` snapshot survives GDPR erasure.
- **`JwtIssuer`** — HS256 access tokens (15min, `sub/iat/exp/iss`). Algorithm
  confusion attacks rejected at decode (`Validation::new(Algorithm::HS256)`).
  Secret + refresh HMAC secret both wrapped in `secrecy::SecretString`.
- **`Credential::Internal.password: SecretString`** — manual `Debug` redaction,
  zeroed on drop. Breaking change rippled through 391a tests.
- **`POST /v1/auth/login` endpoint** in `garraia-gateway` under feature
  `auth-v1`. Returns 200 `{user_id, access_token, expires_at}` on success,
  401 `{"error":"invalid_credentials"}` byte-identical on every failure
  mode (anti-enumeration validated by integration test).

### Reduced-scope deferrals to GAR-391c

Two structural gaps surfaced empirically during 391b implementation. Both
were resolved by deferring features rather than broadening the
`garraia_login` role:

| Deferral | Why | Lands in |
|---|---|---|
| `refresh_token` in login response | `garraia_login` lacks `SELECT ON sessions`, required by `INSERT ... RETURNING id` and `verify_refresh` | 391c (with refresh endpoint + migration 010) |
| `SessionStore::issue` called by endpoint | same | 391c |
| `InternalProvider::create_identity` real impl | `garraia_login` lacks `INSERT ON user_identities` | 391c (with signup endpoint + signup pool design) |
| Axum `Principal` extractor + `RequirePermission` | by design | 391c |
| Refresh / logout / signup endpoints | by design | 391c |

`SessionStore` and `RefreshTokenPair` types still live in the crate as
load-bearing scaffolding for 391c. They are exercised by unit tests but
not wired to the endpoint in 391b.

## Pool boundaries

| Pool | Role | RLS | Allowed callers |
|---|---|---|---|
| `garraia-workspace::Workspace` | `garraia_app` | enforced | every request handler |
| `garraia-auth::LoginPool` | `garraia_login` | bypassed | login endpoint only (391b) |

Mixing the two is a compile-time error: `LoginPool` is the only constructor
path for a credential-verification-capable pool, and it refuses any role
other than `garraia_login`.

## What ships in 391c

### RBAC primitives

- **`Role` enum** — 5 variants (`Owner`, `Admin`, `Member`, `Guest`, `Child`)
  mirroring `group_members.role` and the `roles` seed in migration 002.
  Tiers: `owner=100, admin=80, member=50, guest=20, child=10`. Round-trip
  stable via `from_str` / `as_str`.
- **`Action` enum** — 22 variants covering every `permissions.id` row
  seeded in migration 002 (`files.*`, `chats.*`, `memory.*`, `tasks.*`,
  `docs.*`, `members.manage`, `group.{settings,delete}`, `export.{self,group}`).
- **`fn can(principal, action) -> bool`** — central RBAC gate. Returns
  `false` if `principal.role.is_none()`. Backed by a 5×22=110-case
  table-driven unit test (`can_matrix_matches_seed`) asserting exactly 63
  trues across every role × action combination.

### Extractors

- **`Principal` (implements `FromRequestParts<S>`)** — extracts
  `Authorization: Bearer <jwt>`, verifies via `JwtIssuer::verify_access`,
  then optionally resolves `X-Group-Id` header into a typed `Role` via
  `SELECT role FROM group_members WHERE group_id=$1 AND user_id=$2 AND
  status='active'` (uses `LoginPool` BYPASSRLS, requires migration 010
  Gap C grant). Returns 401 / 403 / 400 per documented contract.
- **`RequirePermission` is NOT a `FromRequestParts` extractor** — Axum's
  const generics over enums are unstable. Instead, call
  `RequirePermission::check(&principal, action)?` or the free function
  `require_permission(&principal, action)?` inside handlers as inline
  guards returning `(403, "forbidden")` on failure.

### Signup pool

- **`SignupPool` newtype** — dedicated `garraia_signup` BYPASSRLS pool.
  Constructor validates `SELECT current_user = 'garraia_signup'`.
  Non-`Clone` enforced via `static_assertions`. Identical boundary
  contract as `LoginPool`.
- **`signup_user(login_pool, signup_pool, email, password, display_name)`**
  free function — transactional: `INSERT INTO users + user_identities` with
  fresh Argon2id hash. Translates Postgres SQLSTATE 23505 to
  `AuthError::DuplicateEmail`.

### Other additions

- **`RedactedStorageError`** — newtype over `sqlx::Error` whose `Display`
  strips `postgres://` substrings and `host=`/`password=`/`user=` segments.
  Closes the M-3 deferral from 391b.
- **`AuthConfig` (in `garraia-config`)** — 4 env vars: `GARRAIA_JWT_SECRET`,
  `GARRAIA_REFRESH_HMAC_SECRET`, `GARRAIA_LOGIN_DATABASE_URL`,
  `GARRAIA_SIGNUP_DATABASE_URL`. All fields `SecretString` with manual
  `Debug` redaction. Fail-soft (`from_env` returns `Ok(None)` when any
  var missing) + strict variant (`require_from_env`) for prod.
- **4 endpoints wired default-on** (feature flag `auth-v1` REMOVED):
  - `POST /v1/auth/login` — now includes `refresh_token` (Gap A fix).
  - `POST /v1/auth/refresh` — rotation default ON (revoke old before issue new).
  - `POST /v1/auth/logout` — idempotent 204 always.
  - `POST /v1/auth/signup` — uses `SignupPool`; 201+tokens on success, 409 on duplicate.
- **Migration 010** (`010_signup_role_and_session_select.sql` in `garraia-workspace`):
  - Creates `garraia_signup NOLOGIN BYPASSRLS` (Gap B).
  - `GRANT SELECT ON sessions TO garraia_login` (Gap A).
  - `GRANT SELECT ON group_members TO garraia_login` (Gap C).

### Structural gaps resolved

| Gap | Root cause | Resolution |
|-----|-----------|------------|
| A — sessions SELECT | `INSERT ... RETURNING id` and `verify_refresh` need SELECT on sessions | `GRANT SELECT ON sessions TO garraia_login` (migration 010) |
| B — signup INSERT | Login role's scope precludes INSERT on user_identities | New `garraia_signup` role + `SignupPool` newtype (migration 010) |
| C — group_members SELECT | `Principal` extractor membership query failed at runtime | `GRANT SELECT ON group_members TO garraia_login` (migration 010, mid-execution Option 1 approval) |

## Tests

```
cargo test -p garraia-auth
cargo test -p garraia-gateway --features auth-v1 --test auth_v1_login
```

- **16 unit tests** in `src/{hashing,jwt,login_pool}.rs::tests`: PHC params,
  algorithm confusion guards, secret length validation, redacted Debug.
- **3 skeleton integration tests** in `tests/skeleton.rs`: `LoginPool`
  constructor positive + negative + `InternalProvider` shape regression.
- **9 verify integration tests** in `tests/verify_internal.rs`: Argon2id
  happy path, PBKDF2 lazy upgrade happy path, wrong password, user not
  found, suspended, deleted, unknown hash format, find_by_provider_sub,
  create_identity deferred-to-391c marker.
- **1 race regression test** in `tests/concurrent_upgrade.rs`: 5 concurrent
  PBKDF2 verifies for the same user → exactly 1 upgrade audit row, 5
  success rows. Empirical proof that `FOR NO KEY UPDATE OF ui` works.
- **3 endpoint integration tests** in `garraia-gateway/tests/auth_v1_login.rs`:
  happy path 200, byte-identical 401 across 5 failure modes, 422 on type-mismatched JSON.

Each integration test spins its own `pgvector/pgvector:pg16` testcontainer.
Total wall time ~3 minutes on a warm Docker cache. Shared-fixture
optimization (`tokio::sync::OnceCell`) deferred to 391c.
