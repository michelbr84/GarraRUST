## 1. Goal (one sentence)

Adicionar a coluna `hash_upgraded_at timestamptz NULL` em `user_identities` via migration `009_user_identities_hash_upgraded_at.sql`, forward-only e estritamente aditiva, para suportar o lazy upgrade PBKDF2→Argon2id que será implementado em GAR-391b — sem tocar em nenhum código Rust de auth nesta rodada, mantendo a regra de **separar correção estrutural de implementação de feature**.

---

## 2. Rationale — por que essa fatia primeiro

1. **Wave 0 reconnaissance de 391b descobriu o gap.** A coluna não existe em migration 001 (verificado por leitura direta de `crates/garraia-workspace/migrations/001_initial_users_groups.sql` linhas 46-54).
2. **Regra explícita do usuário (2026-04-14):** "Não misture correção estrutural com implementação de auth no mesmo impulso. Se faltar mais do que o aceitável, pare e me entregue o plano separado da migration 009/0011.5 antes de continuar." Mesmo sendo um único `ADD COLUMN`, a regra se aplica.
3. **Risco de bundling:** se a migration entrar no mesmo commit que o `verify_credential` real, um eventual `git revert` para corrigir um bug do crypto path forçaria também a remoção da coluna — mas migrations são forward-only e não devem ser revertidas. Separar mata esse risco em definitivo.
4. **Tempo de delivery:** plan 0011.5 é cirúrgico (≤1h). Custo de separação ≈ zero comparado ao benefício de honrar a regra arquitetural.
5. **Evidence trail no Linear:** a migration corretiva fica registrada como entrega independente, fácil de auditar separadamente da feature de auth.

## 6. Schema details

```sql
-- 009_user_identities_hash_upgraded_at.sql
-- GAR-391b prereq — Adiciona user_identities.hash_upgraded_at para suportar
-- o lazy upgrade PBKDF2→Argon2id no verify_credential do crate garraia-auth.
--
-- Plan:     plans/0011.5-gar-391b-migration-009-hash-upgraded-at.md
-- ADR:      docs/adr/0005-identity-provider.md (§"InternalProvider implementation outline")
-- Depends:  migration 001 (user_identities table).
-- Forward-only. Strictly additive nullable column. No DROP, no destructive ALTER,
-- no backfill. NULL = "this row has never been rotated".
--
-- ─── Why a separate migration instead of folding into 391b ────────────────
-- User rule (2026-04-14): structural corrections discovered during Wave 0
-- reconnaissance must NOT be bundled with the feature implementation that
-- needs them. Migration 009 ships alone, then GAR-391b lands on top.
--
-- ─── Caller responsibility ────────────────────────────────────────────────
-- Only the lazy upgrade path in InternalProvider::verify_credential
-- (garraia-auth, GAR-391b) writes to this column. Production writes happen
-- inside the same transaction as the UPDATE password_hash, under a
-- SELECT ... FOR NO KEY UPDATE OF ui row lock. Any other writer is a bug.

ALTER TABLE user_identities
    ADD COLUMN IF NOT EXISTS hash_upgraded_at timestamptz NULL;

COMMENT ON COLUMN user_identities.hash_upgraded_at IS
    'Timestamp of the last password_hash algorithm rotation (e.g., PBKDF2 → Argon2id '
    'lazy upgrade). NULL means the hash has never been rotated since the row was '
    'inserted. Written ONLY by garraia-auth::InternalProvider::verify_credential '
    'inside the lazy upgrade transaction. See ADR 0005 §"InternalProvider implementation '
    'outline" and plan 0011 (GAR-391b).';
```

## 8. Rollback plan

Migrations são forward-only, então "rollback" formal não existe. Cenários reais:

1. **Antes do merge:** fechar o PR. Trivial.
2. **Após o merge, antes de 391b shippar:** a coluna fica no DB sem callers. Inerte. Custo = 8 bytes por linha de `user_identities` em qualquer instância que rodou a migration. Aceitável.
3. **Após 391b shippar:** a coluna é load-bearing. Forward-fix only — qualquer mudança vira nova migration.

Zero risco de data loss porque o ADD COLUMN é puramente aditivo e nullable.

## 9. Risks & mitigations

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| `ADD COLUMN` em tabela grande causa lock acquisition lento | Low (DB de dev/teste é vazio; produção self-host tem dezenas de linhas no máximo nesta fase) | Low | Postgres 11+ faz `ADD COLUMN ... NULL` sem default como operação O(1) (apenas metadata change). Sem rewrite de tabela. |
| Outra migration futura tenta usar o mesmo número 009 | Negligible | Low | Convenção do projeto é numérica monotônica; collision seria pego em code review. |
| Smoke test ficou frágil ao tipo retornado por information_schema | Low | Low | `data_type = 'timestamp with time zone'` é o nome canônico do Postgres para `timestamptz`. Estável entre versões. |
| Plan 0011 (391b) precisar de mais colunas do que esta migration entrega | Low (Wave 0 só achou esta) | Medium | Se 391b descobrir nova falta DENTRO do Wave 1, parar de novo e abrir migration 010 com plan 0011.6. Mesma regra. |
| Bundling acidental de código Rust junto | Negligible | Medium | Plano §3 é explícito: ZERO Rust nesta rodada. Code reviewer flag-fica qualquer arquivo Rust no diff. |
